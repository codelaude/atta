# CI Review — Setup & Customization

> Atta's context-aware CI review for GitHub pull requests.
> Generated by `atta init --adapter github-action`.

---

## How It Works

The generated workflow (`.github/workflows/atta-review.yml`) runs an AI-powered
code review on every PR. Unlike generic CI review actions, it reads your project's
own knowledge files before reviewing — so it knows your tech stack, conventions,
and which security checks actually apply.

**What the CI reviewer reads:**
- `.atta/project/project-context.md` — tech stack and architecture
- `.atta/project/project-profile.md` — team conventions and review priorities
- `.atta/team/patterns/*.md` — technology-specific conventions
- `.atta/team/rules/` — path-scoped coding rules (v3.0)
- `.atta/team/owasp-scope.md` — pre-computed security scope for your stack (v3.0)
- `.atta/team/ci-suppressions.md` — known false positives to skip

**Result:** Fewer false positives, findings scoped to your actual stack, and
convention violations caught before human review.

## Supported Providers

The CI adapter supports multiple AI providers via `--provider` and `--auth-backend` flags:

| Provider | Flag | Secret Required | Notes |
|----------|------|-----------------|-------|
| **Anthropic** (default) | `--provider anthropic` | `ANTHROPIC_API_KEY` | Uses `claude-code-action` |
| **AWS Bedrock** | `--auth-backend bedrock` | AWS credentials | Claude via Bedrock |
| **GCP Vertex** | `--auth-backend vertex` | GCP credentials | Claude via Vertex AI |
| **Azure Foundry** | `--auth-backend foundry` | Azure credentials | Claude via Azure |
| **OpenAI** | `--provider openai` | `OPENAI_API_KEY` | Uses `LLM-action` (OpenAI-compatible) |
| **Azure OpenAI** | `--provider azure` | Azure OpenAI credentials | Uses `LLM-action` |
| **Ollama** | `--provider ollama` | None (self-hosted) | Uses `LLM-action` |

```bash
# Default (Anthropic direct)
atta init --adapter github-action

# AWS Bedrock
atta init --adapter github-action --auth-backend bedrock

# OpenAI
atta init --adapter github-action --provider openai

# Self-hosted Ollama
atta init --adapter github-action --provider ollama
```

## Setup

1. **Add the secret**: Go to `Settings → Secrets → Actions` in your repo.
   Add the appropriate API key for your chosen provider (e.g., `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`).

2. **Commit the workflow**: The workflow file is not auto-committed.
   ```bash
   git add .github/workflows/atta-review.yml .atta/team/ci-suppressions.md
   git commit -m "feat: add Atta CI review workflow"
   ```

3. **Run `/atta` first** (recommended): If you haven't already, run `/atta` in your
   project to detect your tech stack and generate pattern files. This significantly
   improves review quality — without pattern files, the CI falls back to generic checks.

## Improving Review Quality

The CI reviewer is only as good as your knowledge files. In rough order of impact:

| File | How to Populate | Impact |
|------|----------------|--------|
| `project-context.md` | Run `/atta` | High — scopes security checks to your stack |
| `patterns/*.md` | Run `/atta` or create manually | High — teaches conventions |
| `team/rules/` | Auto-generated by `/atta` (v3.0) | High — path-scoped coding rules |
| `project-profile.md` | Run `/atta-profile` or edit manually | Medium — team conventions and review priorities (committed) |
| `ci-suppressions.md` | Add manually over time | Medium — reduces noise |

## Suppression Workflow

When CI flags something that isn't a real issue:

1. **Verify it's a false positive** — paste the finding into your local AI tool to confirm
2. **Add a suppression entry** to `.atta/team/ci-suppressions.md` in your PR branch
3. **Human reviewer approves** the suppression in the PR diff (or rejects it)
4. On merge, all future PRs benefit from the suppression

Over time, use `/atta-patterns promote` to replace raw suppressions with understood patterns.
The AI stops flagging the whole category instead of just the specific instance.

## Customizing the Workflow

The workflow YAML structure (runner, permissions, action version, branch filters)
is yours to configure. Atta owns the `prompt:` section.

To update the prompt after major convention changes:
```bash
atta init --adapter github-action -d /path/to/project
```

This regenerates the workflow file. Your `ci-suppressions.md` is never overwritten.

**Common customizations:**

```yaml
# Only review PRs to main/master (not feature branches)
on:
  pull_request:
    branches: [main, master]
    types: [opened, synchronize, reopened]

# Add timeout to cap costs
jobs:
  review:
    timeout-minutes: 10
```

## Architecture

```
PR opened/updated
       │
       ▼
GitHub Action runs
       │
       ▼
AI reads .atta/team/ files              ← project knowledge (committed to repo)
       │
       ▼
Context-scoped review of changed files
       │
       ▼
PR comment with findings
       │
       ▼
Author triages findings
  ├── Real issue → fix in PR
  └── False positive → add to ci-suppressions.md in same PR
                              │
                              ▼
                     Human reviewer approves suppression
                              │
                              ▼
                     Merges to main → all future PRs benefit
```

**Key design decisions:**
- **Read-only CI**: the action never writes to `.atta/` — learning stays local
- **Human-gated suppressions**: every suppression lands in a PR diff for review
- **Knowledge files as source of truth**: context is committed, not embedded in YAML
- **Provider-agnostic**: works with Anthropic, OpenAI, Azure, Bedrock, Vertex, or self-hosted Ollama
