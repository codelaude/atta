# Path-Scoped Coding Rules

**Added in**: v3.0.0

## Overview

Path-scoped rules are tech-specific coding standards generated from your detected stack and placed in each adapter's native format. They give your AI tool (and CI reviewer) enforceable conventions — not just documentation, but rules applied automatically when editing matching files.

## How It Works

1. **`/atta` detects your stack** — React, TypeScript, PostgreSQL, etc.
2. **Rules generated to `.atta/team/rules/`** — a single tool-agnostic source of truth
3. **Adapter formats the rules** — each tool gets rules in its native discovery path

| Adapter | Output Path | Format |
|---------|-------------|--------|
| Claude Code | `.claude/rules/atta-*.md` | Markdown with `paths:` frontmatter for file scoping |
| Copilot | `.github/instructions/*.instructions.md` | Markdown with `applyTo:` globs |
| Cursor | `.cursor/rules/*.mdc` | MDC format with `globs:` frontmatter |
| Gemini | `.gemini/styleguide.md` | Concatenated natural-language rules |
| Codex | `AGENTS.md` appendix | Inline in the instruction file |
| CI (GitHub Action) | Prompt context | Read from `.atta/team/rules/` at review time |

## Source of Truth

All rules live in `.atta/team/rules/` as tool-agnostic markdown files. Each file covers one technology:

```
.atta/team/rules/
├── react-rules.md            # React component patterns, hooks rules
├── typescript-rules.md       # Strict typing, generics, error handling
├── express-rules.md          # Middleware, routing, error handling
├── postgresql-rules.md       # Query optimization, indexing, transactions
└── security-rules.md         # OWASP-scoped checks for your stack
```

The adapter then copies and reformats these into the tool's native path during `npx atta-dev init`.

## Customizing Rules

### Add a rule

Edit the relevant file in `.atta/team/rules/`:

```markdown
## Naming

- Use PascalCase for React components
- Use camelCase for hooks (prefix with `use`)
- Use SCREAMING_SNAKE_CASE for constants
```

Then re-run init to propagate to your adapter:
```bash
npx atta-dev init --update
```

### Remove a rule

Delete the line from `.atta/team/rules/` and re-run init.

### Project-specific rules

Add a new file to `.atta/team/rules/` with your project conventions:

```markdown
# Project Rules

## API Conventions
- All REST endpoints return `{ data, error, meta }` envelope
- Use `zod` for request validation, never manual parsing
- Rate limiting on all public endpoints
```

## How Rules Differ from Patterns

| | Rules (`.atta/team/rules/`) | Patterns (`.atta/team/patterns/`) |
|---|---|---|
| **Purpose** | Enforced conventions | Reference documentation |
| **Format** | Short, actionable statements | Detailed examples with DO/DON'T |
| **Loaded by** | Tool's native rule system (always in context) | Agents on demand |
| **Scope** | File-type specific (via globs) | Technology-wide |
| **Generated by** | `/atta` from detection + mappings | `/atta` from pattern templates |

Rules are what the tool enforces. Patterns are what agents reference when reviewing or guiding.

## CI Integration

The CI review adapter reads rules directly from `.atta/team/rules/` — no separate generation step. This means your CI review uses the same conventions as your local tool.

See [CI Review](ci-review.md) for the full CI setup.
