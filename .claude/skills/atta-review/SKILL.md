You are now acting as the **Code Reviewer** with automated pattern checking capabilities.

## How to Use

```
/atta-review                              # Review recent changes (git diff)
/atta-review src/components/UserProfile.tsx  # Review specific file
/atta-review src/components/search          # Review folder
/atta-review --quick src/auth/login.ts      # Quick review (critical only)
```

---

## Execution Steps

### Step 0: Load Scoped Directives

Read these files from `.claude/agents/memory/` if they exist (skip silently if absent):
- `directives-code-reviewer.md` — code review rules
- `directives-style.md` — style and formatting rules

Apply any directives found as additional review constraints.

### Step 0b: Load Review Guidance (if present)

Load stack-specific review rules generated by Atta during init. These are **additive** — they add tech-specific checks alongside Steps 3-4, they do not replace the built-in checks.

**Primary** — read all `.atta/team/rules/*.md` files if the directory exists:
- Each file contains Key Rules and Anti-Patterns for a detected technology
- Apply every loaded rule as an additional review constraint during Steps 3-4
- If a loaded rule overlaps with a built-in check, note both but do not double-count

**Fallback** — if `.atta/team/rules/` is empty or missing, read `REVIEW.md` (repo root):
- Contains the same data formatted as review guidance (severity-filtered)

**Skip silently** if neither source exists — the skill works standalone with built-in checks only.

### Step 1: Determine Review Scope

**If no argument provided:**
```bash
# Detect base branch dynamically: main, master, or develop (whichever exists)
if git rev-parse --verify --quiet origin/main >/dev/null 2>&1; then
  FILES=$(git diff --name-only origin/main...HEAD)
elif git rev-parse --verify --quiet origin/master >/dev/null 2>&1; then
  FILES=$(git diff --name-only origin/master...HEAD)
elif git rev-parse --verify --quiet origin/develop >/dev/null 2>&1; then
  FILES=$(git diff --name-only origin/develop...HEAD)
else
  # No remote base found — fall back to uncommitted changes
  FILES=$(git diff --name-only)
fi
```
Review all files in `$FILES`. If `$FILES` is empty, trigger the "Empty Review Scope" recovery.

**If file/folder argument:**
Read the specified target.

### Step 2: Auto-Read Files

For each file in scope, use the Read tool to load the content. Don't ask the user - just read them.

### Step 3: Run Automated Checks

Apply these universal pattern checks automatically. Tech-specific checks come from Step 0b rules — apply those alongside these baseline patterns.

#### CRITICAL Checks (Auto-Fail)

| Pattern | Check |
|---------|-------|
| `eval()` / `Function()` with dynamic input | Code injection risk |
| Unsafe HTML injection (`innerHTML`, `v-html`, `dangerouslySetInnerHTML`) | XSS vector |
| Hardcoded credentials (`password=`, `secret=`, `api_key=`, `token=`) | Secrets leak |
| Disabled security controls (`verify=False`, `csrf: false`, `--no-verify`) | Security bypass |

#### HIGH Checks

| Pattern | Check |
|---------|-------|
| Non-semantic interactive elements (`<div>` with click/key handlers) | Should be `<button>` or `<a>` |
| Timers/subscriptions/listeners without cleanup | Resource leak |
| Debug statements left in code (`console.log`, `print`, `debugger`, `TODO`, `FIXME`) | Cleanup before merge |
| Catch blocks that swallow errors silently | Missing error handling |

### Step 4: Domain-Specific Review

After automated checks, review for these language-agnostic concerns. For tech-specific checks (framework patterns, language idioms, styling conventions), apply rules loaded in Step 0b.

#### Module Structure
- [ ] Idiomatic patterns for the project's framework/language
- [ ] Public API surface explicitly defined
- [ ] Inputs/outputs properly typed
- [ ] Naming conventions match project standards

#### Accessibility
- [ ] Semantic HTML elements
- [ ] ARIA attributes where needed
- [ ] Keyboard navigation works
- [ ] Focus management on dynamic content

#### Testing
- [ ] Test file exists for changed code
- [ ] Error and edge cases covered
- [ ] Async tests handle timing correctly
- [ ] Cleanup after each test (no leaked state)

#### Security
> Deep security analysis is handled by `/atta-security-audit`. Preflight invokes it automatically.
> During code review, only flag obvious security issues encountered while reading.

#### Performance
- [ ] No expensive computations in hot paths
- [ ] Large lists use pagination or virtualization
- [ ] No synchronous blocking operations in async contexts
- [ ] Event handlers debounced/throttled where needed
- [ ] No N+1 query patterns in data fetching
- [ ] Bundle impact considered (no large imports for small features)

#### Bug & Logic Review
- [ ] Edge cases handled (empty collections, null values, boundary conditions)
- [ ] Error paths don't silently swallow failures
- [ ] Async operations have proper error handling
- [ ] State mutations are intentional (no accidental side effects)
- [ ] Conditional logic covers all branches
- [ ] Loop termination conditions are correct
- [ ] Type coercion doesn't produce unexpected results
- [ ] Race conditions between async operations considered

Categorize each finding by severity: **CRITICAL** / **HIGH** / **MEDIUM** / **LOW**.

### Step 5b: Log Anti-Pattern Findings (Silent)

For each **CRITICAL** or **HIGH** finding, log to the pattern detection system:

```bash
bash .atta/scripts/pattern-log.sh {attaDir} << 'PAYLOAD'
{"category":"anti-pattern","pattern":"<slugified-check-name>","description":"<finding-description>","context":{"file":"<file:line>","domain":"<domain>","agent":"code-reviewer"},"source":"skill-annotation","skill":"review","agentId":"code-reviewer"}
PAYLOAD
```

After logging all findings, run: `bash .atta/scripts/pattern-analyze.sh {attaDir}`

> Skip if no CRITICAL/HIGH findings or if `pattern-log.sh` is not available.

## Review Verdicts

### APPROVED
All checks pass. No critical or high issues. Optional suggestions only.

### CHANGES REQUESTED
Has critical or high-priority issues that must be addressed.

### NEEDS DISCUSSION
Architectural concerns or trade-offs that need user input.

## Quick Review Mode

When `--quick` flag is used:
- Only run CRITICAL automated checks
- Skip domain-specific deep review
- Output pass/fail summary only

## Integration

After `/atta-review`:
- Run `/atta-security-audit` for deep security analysis (OWASP Top 10, deps, secrets)
- Run `/atta-lint` for pattern-only checks
- Run `/atta-preflight` for full pre-PR validation
- Use `/atta-collaborate` for multi-agent cross-domain review

### Multi-Agent Collaboration

When the review scope spans **multiple domains** (e.g., component files + API routes + styles), suggest:

```markdown
---
**Multi-Agent Collaboration Available**

This review touched files across multiple domains ([list domains]).
For deeper cross-domain analysis with automated conflict detection, run:

`/atta-collaborate`                           # Auto-routes to relevant specialists
`/atta-collaborate --agents security-specialist,accessibility`  # Explicit agent selection
```

**When to suggest** (any of these conditions):
- Review scope includes 2+ file type categories
- Security findings were flagged alongside framework findings
- Accessibility concerns overlap with component structure issues

**When NOT to suggest**: Single-file review, all same type, or `--quick` mode.

## Error Handling & Recovery

### Cannot Determine Diff Scope

```markdown
Could not resolve a review diff against the base branch.

Recovery options:
1. Review local changes only (`git diff --name-only`)
2. Provide a file/folder target explicitly (`/atta-review path/to/file`)
3. Fetch remotes and retry when `origin/main|master|develop` is available
```

### Empty Review Scope

```markdown
No files found to review.

Recovery options:
1. Pass a target explicitly (`/atta-review src/components/search`)
2. Stage or modify files, then rerun `/atta-review`
3. Use `/atta-review --quick <file>` for a targeted critical-pass check
```

### Agent Context Missing

```markdown
Review context is incomplete (missing generated agents or patterns).

Recovery options:
1. Run `/atta` to generate missing agents/pattern files
2. Continue with baseline critical checks only
3. Route to `/atta-agent code-reviewer` for a manual focused review
```
