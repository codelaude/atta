# Agent Routing Protocol

Operational guidance for routing decisions. Loaded when deciding which agent or specialist to delegate work to.

## Core Agents (always available)

| Agent | Role | When to invoke |
|-------|------|----------------|
| project-owner | Orchestrator — routes tasks, synthesizes multi-agent output | Complex features, cross-domain work, task decomposition |
| code-reviewer | Code quality, patterns, security | Code review, pattern checks, pre-PR review |
| librarian | Knowledge capture, directives, corrections | "Remember to...", "Always...", "Never...", corrections |
| business-analyst | Requirements, acceptance criteria, edge cases | Clarifying business logic, defining ACCs |
| qa-validator | ACC validation, test scenario creation | Checking if code meets requirements |
| pr-manager | PR descriptions, commit messages, completion criteria | Preparing PRs, writing commit messages |
| rubber-duck | Guided learning, problem exploration | When user wants to think through a problem |

## Generated Agents (project-specific)

Running `/atta` detects the project's tech stack and generates specialists. These vary per project:

- **Team leads** (e.g., fe-team-lead, be-team-lead) — coordinate domain specialists
- **Framework specialists** — idiomatic patterns for the detected framework
- **Language specialists** — best practices for the detected language
- **Domain specialists** — styling, accessibility, database, security, testing, E2E

Use `/atta-agent <agent-id>` to invoke a specialist. To discover available agents, check the agents directory generated by `/atta`.

## Routing Decision Tree

```
Is it a code review?
  → /atta-agent code-reviewer (or /atta-review for comprehensive review)

Is it a specific specialist domain?
  → /atta-agent <specialist-id>

Does it touch multiple domains?
  → /atta-agent project-owner decomposes, delegates to specialists in parallel

Is it a complex feature that needs decomposition?
  → /atta-team-lead (breaks into specialist-specific tasks)

Does the change need multi-perspective review? (2+ domains)
  → /atta-collaborate (auto-routes to 2-4 specialists)

Is it a knowledge capture? ("remember...", "always...", "never...")
  → /atta-agent librarian

Is it requirements/acceptance criteria?
  → /atta-agent business-analyst

Is it PR preparation?
  → /atta-agent pr-manager (or /atta-ship for full workflow)

Is it a quick single-domain task?
  → /atta-agent <id> (skip coordination overhead)
```

## Multi-Agent Collaboration

`/atta-collaborate` auto-routes to specialists based on file patterns detected in the project. It selects 2-4 agents, prioritizing by: **security > accessibility > framework > language > styling > testing > database > code-reviewer**.

Cross-cutting rules (always included when triggered):
- **Security specialist**: when scope touches auth, credentials, config, or code injection patterns
- **Accessibility specialist**: when scope touches UI components

## Parallel vs Sequential

**Parallel** — independent tracks, multiple specialists reviewing from different angles
**Sequential** — dependent work where one agent's output informs the next

## Escalation to User

Escalate when:
- Conflicting recommendations between agents
- Business decisions needed (scope, prioritization)
- Security implications affecting architecture
- Breaking API changes
- Ambiguous requirements

## Agent Lookup

When loading an agent by ID, `/atta-agent` searches the project's agents directory in order: core agents (root), then coordinators, then specialists. The exact directory path depends on which tool is running (each adapter installs agents to the tool's native location).
