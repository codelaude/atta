You are running **migration** from a previous `.claude/` version to v2.0 (Bootstrap System).

## How to Use

```
/migrate                       # Migrate to v2.0
/migrate --rollback            # Restore from backup
/migrate --add-update-system   # Add update tracking to existing v2.0 project
```

---

## What This Skill Does

1. **Detect** the current version and structure
2. **Back up** the existing `.claude/` directory
3. **Restructure** agents into the three-tier system
4. **Preserve** custom rules, directives, and knowledge
5. **Run `/init`** to generate new agents from templates

---

## Phase 1: Detect Current Version

Check for version indicators:

1. **Read `.claude/.metadata/version`** — if exists, already v2.0 (skip migration)
2. **Check for bootstrap directory** — if `.claude/bootstrap/` exists, already v2.0
3. **Check agent structure**:
   - Flat agents at `.claude/agents/` root with tech-specific names (vue.md, scss.md, typescript.md) → v1.0
   - Person-named agents (petra.md, lory.md, etc.) → pre-v1.0
   - No agents directory → fresh install (redirect to `/init`)

### Version Detection Rules

| Indicator | Version |
|-----------|---------|
| `.metadata/version` = "2.0" | v2.0 (no migration needed) |
| `bootstrap/` directory exists | v2.0 (no migration needed) |
| Role-named agents + no bootstrap | v1.0 |
| Person-named agents | pre-v1.0 |
| No `.claude/agents/` | Fresh install → use `/init` |

If already v2.0, inform user and exit:
> "Your project is already on v2.0. Run `/init --rescan` to regenerate agents."

---

## Phase 2: Analyze Existing Setup

Read all existing agent files and extract:

1. **Custom rules** — any rules, anti-patterns, or checklists added by the user
2. **Directives** — content from `.claude/agents/memory/directives.md`
3. **Knowledge files** — pattern files, project context, quick-reference
4. **Agent customizations** — modifications to default agent behavior
5. **Settings** — `.claude/settings.local.json` permissions

Build a migration manifest:
```
Detected: v1.0
Agents found: [list of .md files in agents/]
Custom rules: [count per agent]
Directives: [count]
Pattern files: [list]
Knowledge files: [list]
```

Present to user for confirmation before proceeding.

---

## Phase 3: Back Up

Create a timestamped backup:

```
.claude/.backup-v1.0-[YYYYMMDD-HHMMSS]/
├── agents/          (copy of all agent files)
├── knowledge/       (copy of knowledge directory)
├── skills/          (copy of skills directory)
└── settings.local.json
```

Confirm backup was created successfully before continuing.

---

## Phase 4: Restructure

### Move tech-specific agents to legacy

Move any v1.0 tech-specific agents (vue.md, scss.md, typescript.md, tester.md, accessibility.md, etc.) to `.claude/agents/legacy/`:

```
agents/vue.md → agents/legacy/vue.md
agents/scss.md → agents/legacy/scss.md
agents/typescript.md → agents/legacy/typescript.md
agents/tester.md → agents/legacy/tester.md
agents/accessibility.md → agents/legacy/accessibility.md
```

### Keep core agents at root

These stay at `.claude/agents/` root (do not move):
- project-owner.md
- librarian.md
- rubber-duck.md
- code-reviewer.md
- business-analyst.md
- qa-validator.md
- pr-manager.md

### Remove old coordinators

If old coordinator agents exist (fe-team-lead.md, backend-consultant.md), move to legacy:
```
agents/fe-team-lead.md → agents/legacy/fe-team-lead.md
agents/backend-consultant.md → agents/legacy/backend-consultant.md
```

These will be regenerated by `/init` in the coordinators/ directory.

### Create directories

Ensure these directories exist:
- `.claude/agents/coordinators/`
- `.claude/agents/specialists/`
- `.claude/bootstrap/` (should already exist from v2.0 repo)
- `.claude/.metadata/`
- `.claude/skills/generated/`

---

## Phase 5: Preserve Custom Rules

For each legacy agent, extract custom content:

1. Read the legacy agent file
2. Identify user-added rules (anything beyond the default template)
3. Store extracted rules for injection into generated agents

These preserved rules will be passed to `/init` so they can be merged into the newly generated specialists.

---

## Phase 6: Run /init

Invoke the `/init` skill to:
1. Detect the current tech stack
2. Generate new coordinators and specialists from templates
3. Create pattern files
4. Generate INDEX.md
5. Configure MCP servers (if desired)

Pass the preserved custom rules so `/init` can merge them into generated agents.

---

## Phase 7: Write Metadata

Write version file:
```
.claude/.metadata/version → "2.0"
```

Update manifest with migration details.

## Rollback Mode (`--rollback`)

When `--rollback` is used:

1. **Find latest backup**: Look for `.claude/.backup-v1.0-*` directories
2. **Confirm with user**: Show backup timestamp and ask for confirmation
3. **Restore**: Copy backup contents back to `.claude/`
4. **Clean up**: Remove v2.0-specific directories (coordinators/, specialists/, bootstrap/, .metadata/)
5. **Report**: Confirm rollback was successful

## Add Update System to Existing v2.0 (`--add-update-system`)

When `--add-update-system` is used on an **existing v2.0 project**:

This adds the update tracking system to a v2.0 project that was created before the update system existed.

### Prerequisites

- Project must already be v2.0 (has `.claude/.metadata/version` = "2.0")
- Has `.claude/bootstrap/` directory
- Has generated agents (coordinators, specialists)

If not v2.0, run `/migrate` first.

### Step 1: Verify v2.0 Project

```bash
# Check version
if [ ! -f .claude/.metadata/version ] || [ "$(cat .claude/.metadata/version)" != "2.0" ]; then
  echo "⚠️  This is not a v2.0 project. Run /migrate first."
  exit 1
fi

# Check bootstrap exists
if [ ! -d .claude/bootstrap ]; then
  echo "⚠️  Bootstrap directory missing. This doesn't appear to be v2.0."
  exit 1
fi

echo "✓ Verified v2.0 project"
```

### Step 2: Analyze Current Files

Scan `.claude/` directory and classify all files:

1. **Compare against framework template**:
   - Read each file
   - Calculate hash
   - Compare with framework version (if available)
   - Detect customizations

2. **Check generated-manifest.json**:
   - See which files were generated
   - Check for `"customized": true` flag
   - Extract customization details

3. **Identify user files**:
   - `knowledge/project/*` - user content
   - `knowledge/accs/*` - user content
   - `agents/memory/*` - user content
   - `agents-config.json` - user content (if exists)
   - `settings.local.json` - user content (if exists)

### Step 3: Build File Manifest

Create `.claude/.metadata/file-manifest.json`:

```json
{
```

### Detection Algorithm for Customizations

For each core agent file, detect customizations by:

1. **Personality Names**:
   - Look for custom names in agent config or front matter
   - Check if agent is referenced by non-standard name

2. **Modified Sections**:
   - Compare against original template (if available)
   - Identify added/removed/modified sections
   - Common sections: Role, Constraints, Context, Rules

3. **Added Rules**:
   - Look for "Custom Rules" or "Project Rules" sections
   - Check for user-added anti-patterns or checklists

4. **Hash Comparison**:
   - Calculate hash of current file
   - Compare with framework template hash
   - If different → customized
